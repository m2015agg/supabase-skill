Here's the FK chain:

```
auth.users (Supabase Auth)
    │
    │  (no FK constraint, but chat_sessions.user_id stores auth UID)
    ▼
chat_sessions
  - id (PK, uuid)
  - user_id (uuid, NOT NULL) — links to auth user, no DB-level FK
  - is_pinned, is_deleted, title, context_summary, metadata...
    │
    │  chat_messages.session_id → chat_sessions.id
    ▼
chat_messages
  - id (PK, uuid)
  - session_id (uuid, FK → chat_sessions.id)
  - role, content, context_sources, routing_metadata...
```

**Key observations:**

- `chat_sessions.user_id` has **no FK constraint** to `auth.users` — it's a bare UUID. User scoping is enforced by RLS, not a DB constraint.
- `chat_messages.session_id` has a **real FK** to `chat_sessions.id` — messages can't orphan.
- `chat_sessions` is also referenced by `gpt5_debug_logs.session_id` (for tracing AI calls per session).
- Sessions have ~16K rows; messages have ~37K rows (~2.3 messages/session average), consuming 373 MB total.
