Here's the FK chain:

```
auth.users (Supabase Auth)
    │
    │  user_id (no FK constraint — loose reference)
    ▼
bibleai.chat_sessions
    id  ←── session_id (FK)
    │
    ▼
bibleai.chat_messages
```

**Key observations:**

- `chat_sessions.user_id` → `auth.users.id` — **no enforced FK constraint** in the schema snapshot (Supabase auth tables live in a separate schema, so RLS policies enforce this instead)
- `chat_messages.session_id` → `chat_sessions.id` — **proper FK constraint** enforced at the DB level
- `chat_messages` has no direct `user_id` column — you must join through `chat_sessions` to get the user

**To query messages for a user:**
```sql
SELECT m.*
FROM bibleai.chat_messages m
JOIN bibleai.chat_sessions s ON m.session_id = s.id
WHERE s.user_id = '77b036d1-32a2-46f8-b1c6-a17e20752c8c';
```

Also notable: `chat_messages.resolved_by` (uuid) is another loose user reference — it stores the ID of whoever resolved a feedback item, but no FK to auth.users either.
