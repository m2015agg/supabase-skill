Here's the FK chain:

```
auth.users (Supabase auth)
    ↓ (no FK, application-level join on user_id)
chat_sessions
  - user_id: uuid (references auth user, no FK constraint)
  - is_pinned, is_deleted, title, etc.
    ↓ chat_messages.session_id → chat_sessions.id
chat_messages
  - session_id: uuid FK → chat_sessions.id
  - role, content, feedback_type, etc.
```

**Key observations:**

1. **`chat_sessions.user_id`** — no FK constraint to `auth.users`. It's a UUID column that's enforced at the application layer (via RLS policies), not at the DB level.

2. **`chat_messages.session_id`** — hard FK → `chat_sessions.id`. Messages cannot exist without a session.

3. **No direct user link on messages** — to get a user's messages, you must join through sessions: `users → chat_sessions (user_id) → chat_messages (session_id)`.

4. **`chat_messages.resolved_by`** — also a uuid referencing a user, but again no FK constraint.

**Scale:** ~16k sessions, ~37k messages (373 MB — messages dominate storage by far).
