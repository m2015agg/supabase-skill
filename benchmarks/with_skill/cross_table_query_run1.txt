Here's the FK chain:

```
auth.users (Supabase Auth)
    │
    │ user_id (uuid, no FK constraint)
    ▼
chat_sessions  (~16,472 rows, 8.8 MB)
    │  id (PK)
    │
    │ session_id → chat_sessions.id
    ▼
chat_messages  (~37,562 rows, 373 MB)
       id (PK)
```

**Key observations:**

1. **`chat_sessions.user_id`** — uuid pointing to the auth user, but **no FK constraint** defined in the schema. It's a logical relationship only (no referential integrity enforced at DB level).

2. **`chat_messages.session_id → chat_sessions.id`** — proper FK constraint enforced.

3. **No direct user FK on `chat_messages`** — user context flows through the session: `chat_messages → chat_sessions.user_id`.

4. **Side tables hanging off `chat_sessions`:** `gpt5_debug_logs` and `v_routing_analytics` also reference `chat_sessions.id`.

5. `chat_messages` is by far the largest table at **373 MB** vs 8.8 MB for sessions — messages are ~42x heavier per row due to `content text`, `context_sources jsonb`, and `processing_metadata jsonb` columns.
