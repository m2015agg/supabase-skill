Here's the FK chain:

```
auth.users (Supabase Auth)
    ↓  (no enforced FK, but logical join)
bibleai.chat_sessions  (user_id uuid → auth.users.id)
    ↓  chat_messages.session_id → chat_sessions.id  [enforced FK]
bibleai.chat_messages
```

**Key observations:**

- `chat_sessions.user_id` references `auth.users.id` but **no database-level FK constraint** is defined — it's a logical/soft relationship only (common in Supabase when the referenced table is in a different schema, `auth` vs `bibleai`)
- `chat_messages.session_id → chat_sessions.id` is a **hard FK constraint** (enforced)
- `chat_messages` also has `resolved_by uuid` which is another soft user reference (no FK)

**To query the full chain:**
```python
# Sessions for a user
await supabase.schema('bibleai').table('chat_sessions')\
    .select('*, chat_messages(*)')\
    .eq('user_id', user_id)\
    .execute()
```
