## FK Chain: `auth.users` → `bibleai.users` → `chat_sessions` → `chat_messages`

```
auth.users (Supabase auth schema)
    └── id (uuid PK)
          │
          ▼  (implicit — bibleai.users.id mirrors auth.users.id)
bibleai.users
    └── id (uuid PK)
          │
          ▼  (no explicit FK declared in chat_sessions)
bibleai.chat_sessions
    ├── id (uuid PK)
    └── user_id (uuid) ──── references users.id [NO DECLARED FK — bare uuid]
          │
          ▼
bibleai.chat_messages
    ├── id (uuid PK)
    └── session_id (uuid) → chat_sessions.id  [explicit FK]
```

---

### Key observations

| Relationship | FK declared? | Notes |
|---|---|---|
| `chat_messages.session_id → chat_sessions.id` | **Yes** (1 FK) | Hard constraint, enforced by DB |
| `chat_sessions.user_id → users.id` | **No** | Bare `uuid`, no FK constraint in schema |
| `users.id → auth.users.id` | **No** | Convention — `bibleai.users.id` is expected to match Supabase auth user IDs |

The weakest link is `chat_sessions.user_id` — it's unguarded. Orphaned sessions are possible if a user is deleted from `auth.users` without cascading cleanup.
